## 외부 서비스 연동 전 (.env 실제 값 채우기)

현재 `.env`에 placeholder 값으로 되어 있음. 실제 서비스 연동 시 아래 값들을 채워야 함.

| 환경변수 | 시점 |
|---|---|
| `MAIL_USERNAME`, `MAIL_PASSWORD` | 이메일 발송 API 만들 때 (JavaMailSender로 확정, Gmail 앱 비밀번호 사용) |
| `KAKAO_CLIENT_ID`, `KAKAO_CLIENT_SECRET` | 카카오 로그인 API 만들 때 |
| `AWS_ACCESS_KEY`, `AWS_SECRET_KEY`, `S3_BUCKET_NAME` | S3 파일 업로드 API 만들 때 |
| `TOSS_SECRET_KEY` | Toss 결제 API 만들 때 |
| `KAKAO_PAY_CID` | 카카오페이 결제 API 만들 때 |
| `NAVER_PAY_CLIENT_ID`, `NAVER_PAY_CLIENT_SECRET` | 네이버페이 결제 API 만들 때 |


## Q&A

⏺ Swagger 비활성화

Swagger는 API 명세서를 브라우저에서 누구나 볼 수 있게 공개합니다.

개발 중에는 편리하지만, 운영 서버(prod)에서는 문제가 됩니다.

- 어떤 API가 있는지 외부에 노출
- 파라미터 구조, 인증 방식까지 다 보임
- 해커가 공격 포인트를 쉽게 파악 가능

그래서 local/staging 환경에서는 켜고, prod 환경에서는 끄는 것이 일반적입니다.
                                                                                                                                                                                             
---                                                                                                                                                                                        
ddl-auto 변경

현재 application-local.yml에 ddl-auto: update로 설정되어 있습니다.

┌──────────┬───────────────────────────────────────────────────────────────────┐
│    값    │                               동작                                 │
├──────────┼───────────────────────────────────────────────────────────────────┤                                                                                                           
│ update   │ 앱 시작 시 Entity와 DB 테이블을 비교해서 자동으로 컬럼 추가/변경               │                                                                                                           
├──────────┼───────────────────────────────────────────────────────────────────┤                                                                                                           
│ validate │ 앱 시작 시 Entity와 DB 테이블이 일치하는지만 확인, 불일치 시 에러              │                                                                                                           
└──────────┴───────────────────────────────────────────────────────────────────┘

update의 문제점:
- Entity를 잘못 수정해도 Hibernate가 알아서 DB를 바꿔버림
- 실수로 컬럼이 추가/변경될 수 있음
- 운영 DB에서는 절대 사용하면 안 됨

언제 validate로 바꾸나:
- DDL SQL로 직접 테이블을 만들기로 했으므로, 스키마가 안정화되면 validate로 바꿔서 "Entity와 DB가 일치하는지 검증만" 하도록 전환

지금 개발 중에는 Entity 수정이 잦으니 update가 편하고, 나중에 스키마가 확정되면 validate로 바꾸면 됩니다.
