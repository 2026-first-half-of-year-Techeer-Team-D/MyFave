## 1. 중복 인덱스 - `deliveries.orders_id` (오류)

`uq_deliveries_orders_id` UNIQUE 제약조건이 이미 내부적으로 인덱스를 생성하는데, `idx_deliveries_orders_id` 인덱스가 별도로 존재함. 완전히 동일한 인덱스가 2개.

```
-- 중복
ALTER TABLE "deliveries" ADD CONSTRAINT "uq_deliveries_orders_id" UNIQUE ("orders_id");  -- 인덱스 자동 생성
CREATE INDEX "idx_deliveries_orders_id" ON "deliveries"("orders_id");  -- 중복
```

**수정:** `idx_deliveries_orders_id` 제거

## 2. `chat_rooms`에 구매자(상대방) FK 누락 (설계 결함)

C2C 마켓플레이스에서 채팅방은 판매자와 구매자 사이에 생성됨. 현재 `user_id` 1개만 있어 한쪽 참여자만 알 수 있음.

```sql
-- 현재
"user_id" BIGINT NOT NULL  -- 누구? 판매자? 구매자?

-- 권장
"seller_id" BIGINT NOT NULL,  -- 판매자 (products.user_id와 동일)
"buyer_id"  BIGINT NOT NULL   -- 구매자
```

또는 `products.user_id`로 판매자를 유추할 수 있으니 현재 `user_id`를 구매자로 명확히 하고 컬럼명을 `buyer_id`로 변경하는 방법도 있음.

## 3. FK에 `ON DELETE` 전략 미지정 (운영 문제)

모든 FK가 기본값인 `ON DELETE NO ACTION`으로 설정되어 있음. 사용자 탈퇴, 상품 삭제 등의 시나리오에서 FK 위반 에러 발생.

| FK | 현재 | 권장 |
|----|------|------|
| `cart_items.user_id → users` | NO ACTION | CASCADE (탈퇴 시 카트 삭제) |
| `cart_items.product_id → products` | NO ACTION | CASCADE (상품 삭제 시 카트에서 제거) |
| `shipping_addresses.user_id → users` | NO ACTION | CASCADE (탈퇴 시 배송지 삭제) |
| `products.sale_id → sale_events` | NO ACTION | SET NULL (세일 종료 시 연결 해제) |
| `orders.user_id → users` | NO ACTION | RESTRICT (주문 있는 사용자 삭제 방지) |
| `payments.orders_id → orders` | NO ACTION | RESTRICT (결제 있는 주문 삭제 방지) |

**주의:** CASCADE는 연쇄 삭제를 일으키므로 신중하게 적용. 특히 `orders`, `payments`는 RESTRICT로 보호해야 함.

## 4. `short_forms.type` - 예약어 충돌 (ORM 호환성)

`type`은 Java, TypeScript, Python 등 대부분의 언어와 ORM(JPA/Hibernate)에서 예약어이거나 특별한 의미를 가짐.

- JPA `@Inheritance`의 `@DiscriminatorColumn` 기본 이름이 `DTYPE`/`type`
- Hibernate에서 `type` 컬럼명 사용 시 네이밍 충돌 경고 발생 가능

```sql
-- 현재
"type" short_form_type_enum NOT NULL

-- 권장
"display_type" short_form_type_enum NOT NULL
```

## 5. `products.category_code` ENUM 미적용 (일관성)

`condition_code`는 ENUM으로 변환했지만 `category_code`는 아직 `VARCHAR(20)`으로 남아있음.
카테고리가 고정된 목록이라면 ENUM 또는 별도 참조 테이블(`categories`)이 필요.

```sql
-- 카테고리가 자주 추가/변경된다면 → 참조 테이블
CREATE TABLE "categories" (
    "category_code" VARCHAR(20) NOT NULL,
    "category_name" VARCHAR(50) NOT NULL
);
ALTER TABLE "categories" ADD CONSTRAINT "pk_categories" PRIMARY KEY ("category_code");
ALTER TABLE "products" ADD CONSTRAINT "fk_products_categories"
    FOREIGN KEY ("category_code") REFERENCES "categories"("category_code");
```

## 6. `order_items.price` CHECK 누락

`products.price >= 0` CHECK은 있지만 스냅샷인 `order_items.price`에는 CHECK이 없음.

```sql
ALTER TABLE "order_items" ADD CONSTRAINT "chk_order_items_price" CHECK ("price" >= 0);
```

## 7. 쿠폰-주문 연결 부재 (이전 보류건 재확인)

어떤 쿠폰이 어떤 주문에 사용되었는지 추적할 방법이 여전히 없음.
`payments.discount_price`에 금액만 있고 출처(어떤 쿠폰)를 알 수 없음.

**운영상 문제:**
- 쿠폰 사용 내역 조회 불가
- 환불 시 쿠폰 복원 대상 특정 불가
- 쿠폰 남용 감지 불가

## 8. `live_sale_events` VIEW의 `SELECT *` 주의사항

PostgreSQL에서 VIEW의 `SELECT *`는 **생성 시점에 컬럼이 확정**됨. 나중에 `sale_events`에 컬럼을 추가해도 VIEW에 자동 반영되지 않음.

```sql
-- sale_events에 "discount_rate" 컬럼을 추가해도
-- live_sale_events VIEW에는 나타나지 않음
-- → CREATE OR REPLACE VIEW로 재생성 필요
```

## 9. `users` 소프트 삭제 미지원

회원 탈퇴 시 물리 삭제(DELETE)를 하면 FK로 연결된 주문, 결제 이력이 모두 영향받음.
`deleted_at` 패턴으로 논리 삭제를 지원하면 데이터 보존이 가능.

```sql
-- users 테이블에 추가 검토
"deleted_at" TIMESTAMPTZ NULL  -- NULL이면 활성, 값이 있으면 탈퇴
```

이 경우 `uq_users_email` 등 UNIQUE 제약도 부분 인덱스로 변경 필요:
```sql
CREATE UNIQUE INDEX "uq_users_email" ON "users"("email") WHERE ("deleted_at" IS NULL);
```
