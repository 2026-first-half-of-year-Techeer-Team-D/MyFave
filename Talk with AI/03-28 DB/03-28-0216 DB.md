# DB 구조 문제점 분석 (postgresql.example)
> 분석일: 2026-03-28

---

## 1. PK 자동 증가 설정 누락 (심각)

모든 테이블의 PK가 `BIGINT NOT NULL`로만 정의되어 있고 `GENERATED ALWAYS AS IDENTITY` 또는 `SERIAL` 설정이 없음.
애플리케이션 레이어에서 ID를 직접 생성해서 넣어야 하는데, 일반적으로 DB에서 자동 증가를 맡기는 것이 안전함.

```sql
-- 권장
"user_id" BIGINT GENERATED ALWAYS AS IDENTITY NOT NULL
-- 또는
"user_id" BIGSERIAL NOT NULL
```

---

## 2. UNIQUE 제약조건 부재 (심각)

| 테이블 | 컬럼 | 문제 |
|--------|------|------|
| `users` | `email` | 중복 이메일 가입 가능 |
| `users` | `nickname` | 중복 닉네임 허용됨 |
| `users` | `phone` | 중복 전화번호 허용됨 |
| `cart_items` | `(user_id, product_id)` | 동일 상품을 카트에 중복 추가 가능 |
| `orders` | `order_number` | 주문번호 중복 가능 |

```sql
ALTER TABLE "users" ADD CONSTRAINT "uq_users_email" UNIQUE ("email");
ALTER TABLE "users" ADD CONSTRAINT "uq_users_nickname" UNIQUE ("nickname");
ALTER TABLE "cart_items" ADD CONSTRAINT "uq_cart_user_product" UNIQUE ("user_id", "product_id");
```

## 3. 인덱스(INDEX) 전무 (성능)

FK 컬럼들에 인덱스가 하나도 없음. 조회 쿼리 성능에 치명적.

```sql
-- 최소한 아래는 추가 필요
CREATE INDEX idx_products_user_id ON products(user_id);
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_order_items_orders_id ON order_items(orders_id);
CREATE INDEX idx_deliveries_orders_id ON deliveries(orders_id);
CREATE INDEX idx_coupons_user_id ON coupons(user_id);
```

## 4. `products.sale_id` NOT NULL 설계 문제

현재 구조에서는 모든 상품이 반드시 `sale_events`에 속해야 함.
세일 이벤트가 없는 일반 상품 등록이 불가능한 구조. NULL 허용 검토 필요.

```sql
-- 현재
"sale_id" BIGINT NOT NULL  -- 세일 없는 상품 등록 불가

-- 권장
"sale_id" BIGINT NULL  -- 세일 이벤트 없는 일반 상품도 등록 가능
```

## 5. NULL 허용이 부적절한 컬럼들

### `products` 테이블
- `is_soldout`, `created_at`, `updated_at` → NULL 허용인데 NOT NULL이어야 함

### `orders` 테이블
- `order_type`, `order_status`, `created_at`, `updated_at` → NULL 허용. 주문 생성 시점에 반드시 값이 있어야 함

### `payment` 테이블
- `total_product_price`, `total_payment_price`, `payment_status` → 결제 금액과 상태가 NULL이면 데이터 신뢰성 문제

## 6. 테이블 네이밍 컨벤션 불일치

| 문제 | 내용 |
|------|------|
| `payment` (단수) | 다른 테이블은 복수형(`users`, `orders`, `products`)인데 `payment`만 단수 |
| `coupons.id` | 다른 테이블은 `{table_name}_id` 패턴인데 `coupons`만 `id` 사용 |

## 7. ENUM 타입 일관성 부재

일부 상태값은 ENUM 타입으로 정의했지만, 나머지는 VARCHAR + COMMENT로만 처리함.

| 컬럼 | 현재 타입 | 권장 |
|------|----------|------|
| `coupons.status` | VARCHAR(20) | `coupon_status_enum` |
| `coupon_masters.coupon_type` | VARCHAR(20) | `coupon_type_enum` |
| `short_forms.type` | VARCHAR(20) | `short_form_type_enum` |
| `products.condition_code` | VARCHAR(20) | `condition_code_enum` |

---

## 8. 쿠폰-주문 연결 부재

쿠폰이 어떤 주문에 사용되었는지 추적할 방법이 없음.
`payment.discount_price`에 금액은 있지만 어떤 쿠폰을 사용했는지 FK가 없음.

```sql
-- payment 또는 orders 테이블에 추가 검토
"coupon_id" BIGINT NULL REFERENCES "coupons"("id")
```

---

## 9. 저장된 배송지(`shipping_addresses`)와 실제 배송(`deliveries`) 연결 부재

사용자가 저장된 배송지를 선택해서 주문할 때 `deliveries`에 `shipping_id` FK가 없음.
배송지 스냅샷을 그대로 복사하는 구조라면 의도된 것이지만, 어떤 저장 배송지를 사용했는지 추적 불가.

---

## 10. `payment`와 `orders` 결제 재시도 처리 불가

현재 `payment`가 `orders_id`를 FK로 가지며 사실상 1:1 구조.
결제 실패 후 재시도 시나리오를 처리하려면 1:N이 되어야 하는데, UNIQUE 제약이 없어 중복은 되지만 설계 의도가 불명확함.

---

## 11. `chat_rooms` 설계 불완전

- 채팅 메시지를 저장하는 `chat_messages` 테이블이 없음
- 채팅방이 어떤 상품/거래와 연결되는지 FK 없음 (보통 C2C는 상품 기반 채팅)
- `participant_count`를 컬럼으로 관리하면 동기화 문제 발생 가능 (실시간 카운트 계산 권장)

---

## 12. `orders.order_number` 타입 문제

`order_number`가 `INT`인데 주문번호는 날짜+시퀀스 조합 문자열(e.g., `20260328-00001`)로 관리하는 경우가 많음. BIGINT 또는 VARCHAR 검토 필요.
