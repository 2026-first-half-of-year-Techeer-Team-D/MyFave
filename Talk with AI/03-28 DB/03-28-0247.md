# postgresql.example 실행 시 문제점 및 우려사항
> 분석일: 2026-03-28

## 1. 실행 순서 오류 - `orders.final_payment_id` FK (실행 불가)

`orders` 테이블이 `payments` 테이블보다 먼저 생성되는데, `final_payment_id`의 FK는 `payments`를 참조한다.
FK를 ALTER TABLE로 분리해서 선언했기 때문에 테이블 생성 자체는 문제없지만, **FK 추가 순서**에 주의가 필요하다.

현재 파일에서는 `payments` PK가 먼저 선언된 뒤 FK가 선언되므로 **순서상 문제는 없다.**
다만 `DEFERRABLE INITIALLY DEFERRED`가 적용되어 있어, 트랜잭션 내에서만 순환 참조 INSERT가 가능함을 인지해야 한다.

**상태: 통과 (단, 트랜잭션 사용 필수)**

## 2. `created_at` NOT NULL인데 DEFAULT 없는 테이블들 (INSERT 실패 위험)

다음 테이블들의 `created_at`이 `NOT NULL`이지만 `DEFAULT` 값이 없다:

| 테이블 | 영향 |
|--------|------|
| `users` | INSERT 시 created_at 누락하면 에러 |
| `sale_events` | 동일 |
| `products` | 동일 |
| `orders` | 동일 |
| `cart_items` | `created_at`, `updated_at` 둘 다 DEFAULT 없음 |
| `shipping_addresses` | `created_at` DEFAULT 없음 |
| `coupon_masters` | 동일 |
| `coupons` | 동일 |
| `order_items` | 동일 |
| `product_images` | `created_at`, `updated_at` 둘 다 DEFAULT 없음 |
| `chat_rooms` | `created_at` DEFAULT 없음 |
| `short_forms` | 동일 |
| `style_feeds` | 동일 |

**권장:** 백엔드에서 항상 값을 넣어줄 것이 확실하면 문제없지만, 안전을 위해 `DEFAULT CURRENT_TIMESTAMP` 추가 권장.

```sql
-- 예시
"created_at" TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
```

## 3. TIMESTAMP vs TIMESTAMPTZ (타임존 누락)

모든 TIMESTAMP 컬럼이 `TIMESTAMP` (without time zone)으로 선언되어 있다.

- PostgreSQL의 `TIMESTAMP`는 타임존 정보를 저장하지 않음
- 서버 타임존이 바뀌거나, 다른 타임존의 클라이언트가 접속하면 시간이 틀어질 수 있음
- 특히 `sale_start_at`, `sale_end_at` 같은 비즈니스 크리티컬한 시간에 문제 발생 가능

**권장:** `TIMESTAMPTZ` (TIMESTAMP WITH TIME ZONE) 사용

```sql
-- 현재
"created_at" TIMESTAMP NOT NULL
-- 권장
"created_at" TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
```

## 4. `payments` 테이블 금액 컬럼 INT 타입 (오버플로우 우려)

`total_product_price`, `delivery_fee`, `discount_price`, `total_payment_price` 모두 `INT` 타입이다.

- PostgreSQL `INT`는 최대 2,147,483,647 (약 21억)
- 원화 기준 21억원이 최대인데, 고가 상품 대량 주문이나 B2B 확장 시 한계
- 할인 금액 계산 시 음수 처리 등에서 예상치 못한 오버플로우 가능

**우려 수준: 낮음** (일반 C2C에서는 문제없지만 확장 시 `BIGINT` 검토)

---

## 5. `products.price` INT 타입 (소수점 불가)

가격이 `INT`로 선언되어 있어 소수점 단위 가격(예: 할인율 적용 후 반올림 이전 금액)을 저장할 수 없다.

- 원화 기준이면 소수점이 불필요하므로 **현재는 문제없음**
- 다만 추후 외화 결제(달러, 엔화 등) 지원 시 `NUMERIC(12,2)` 전환 필요

**우려 수준: 낮음 (원화 전용이면 무관)**

---

## 6. `deliveries` 테이블 대부분 컬럼이 NULL 허용

`receiver_name`, `receiver_phone`, `receiver_address`, `delivery_status`, `created_at`, `updated_at` 모두 NULL 허용.

- 배송 레코드가 생성되었는데 수신자 이름도, 전화번호도, 주소도 없을 수 있음
- `created_at`이 NULL이면 레코드 생성 시점 추적 불가

**권장:** 최소한 `receiver_name`, `receiver_phone`, `receiver_address`, `delivery_status`, `created_at`은 NOT NULL 검토

---

## 7. `payments.created_at`, `payments.updated_at` NULL 허용

결제 레코드의 생성/수정 시점이 NULL일 수 있다. 결제 감사(audit) 추적에 치명적.

**권장:** NOT NULL + DEFAULT CURRENT_TIMESTAMP

---

## 8. `order_number` UNIQUE 제약에 NULL 허용

```sql
"order_number" VARCHAR(30) NULL
```

PostgreSQL에서 UNIQUE 컬럼이 NULL이면 **여러 행에 NULL이 허용**된다 (SQL 표준).
주문이 생성되었는데 `order_number`가 NULL인 레코드가 다수 존재할 수 있음.

**권장:** 주문번호 생성 로직이 확정되면 NOT NULL로 변경하거나, 생성 시 바로 채워넣는 로직 필수

---

## 9. `GENERATED ALWAYS AS IDENTITY`와 JPA/ORM 호환성

`GENERATED ALWAYS`는 INSERT 시 ID 값을 직접 지정하면 에러가 발생한다.

```sql
-- 이렇게 하면 에러
INSERT INTO users (user_id, email, ...) VALUES (1, 'test@test.com', ...);
-- ERROR: cannot insert a non-DEFAULT value into column "user_id"
```

- Spring Data JPA의 `@GeneratedValue(strategy = GenerationType.IDENTITY)`와는 호환됨
- 하지만 **데이터 마이그레이션, 테스트 데이터 INSERT** 시 `OVERRIDING SYSTEM VALUE` 구문이 필요함

```sql
INSERT INTO users (user_id, email, ...) OVERRIDING SYSTEM VALUE VALUES (1, 'test@test.com', ...);
```

**권장:** 테스트/시드 데이터에서 이 점을 반드시 인지. 또는 `GENERATED BY DEFAULT`로 변경하면 수동 INSERT도 허용됨.

## 10. `sale_events.is_live` 수동 관리 위험

`is_live`가 BOOLEAN 컬럼인데, 실제 라이브 여부는 `sale_start_at`과 `sale_end_at` 사이 현재 시간인지로 판단하는 것이 정확하다.

- `is_live = TRUE`인데 `sale_end_at`이 이미 지난 경우 데이터 불일치
- 배치/스케줄러가 정확히 동기화하지 않으면 사용자에게 잘못된 세일 노출

**권장:** `is_live`를 제거하고 쿼리 시 시간 비교로 계산하거나, 뷰(VIEW)로 제공

```sql
-- 뷰 예시
CREATE VIEW live_sale_events AS
SELECT *, (CURRENT_TIMESTAMP BETWEEN sale_start_at AND sale_end_at) AS is_live
FROM sale_events;
```

## 11. `short_forms`, `style_feeds`에 `updated_at` 컬럼 없음

두 테이블 모두 `created_at`만 있고 `updated_at`이 없다.

- 수정이 절대 발생하지 않는 immutable 데이터라면 문제없음
- 그렇지 않다면 수정 이력 추적 불가

**우려 수준: 확인 필요 (비즈니스 로직에 따라 다름)**

---

## 12. 순환 참조 순서 주의 (orders ↔ payments)

현재 FK 선언 순서:
1. `payments.orders_id → orders.orders_id` (일반 FK)
2. `orders.final_payment_id → payments.payment_id` (DEFERRABLE)

스크립트를 그대로 실행하면 문제없지만, **개별 DDL을 분리 실행**하거나 마이그레이션 도구(Flyway, Liquibase 등)에서 순서가 바뀌면 FK 생성 실패할 수 있다.

**권장:** 마이그레이션 도구 사용 시 이 순서를 명시적으로 관리
