## 1. TIMESTAMPTZ 저장 규칙

모든 시간 컬럼이 `TIMESTAMPTZ`로 변경됨. 백엔드에서 시간을 저장할 때:

- **Java/Spring:** `OffsetDateTime` 또는 `Instant` 사용 (`LocalDateTime` 사용 금지)
- 서버 애플리케이션의 타임존을 `Asia/Seoul`로 고정하거나, UTC로 통일 후 프론트에서 변환

```java
// 좋은 예
@Column(name = "created_at")
private OffsetDateTime createdAt;

// 나쁜 예 - 타임존 정보 손실
@Column(name = "created_at")
private LocalDateTime createdAt;
```

## 2. `created_at` / `updated_at` 처리

DB에 `DEFAULT CURRENT_TIMESTAMP`가 설정되어 있으므로:

- **INSERT 시:** `created_at`을 명시하지 않아도 자동 생성됨
- **UPDATE 시:** `updated_at`은 자동 갱신되지 않음. 백엔드에서 반드시 갱신해야 함

```java
// JPA 사용 시 @PreUpdate로 자동화 권장
@PrePersist
protected void onCreate() {
    this.createdAt = OffsetDateTime.now();
    this.updatedAt = OffsetDateTime.now();
}

@PreUpdate
protected void onUpdate() {
    this.updatedAt = OffsetDateTime.now();
}
```

> `DEFAULT CURRENT_TIMESTAMP`는 INSERT 시점에만 동작한다. UPDATE 시 `updated_at`을 직접 갱신하지 않으면 생성 시점의 값이 영원히 남는다.

## 3. `GENERATED BY DEFAULT AS IDENTITY` PK 사용법

`BY DEFAULT`이므로 ID를 직접 지정해도 에러가 나지 않음.

```java
// JPA 설정
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long userId;
```

**주의:** 테스트/시드 데이터에서 ID를 직접 넣을 경우 시퀀스와 충돌할 수 있음.

```sql
-- 시드 데이터 INSERT 후 시퀀스 동기화 필수
SELECT setval(pg_get_serial_sequence('users', 'user_id'), (SELECT MAX(user_id) FROM users));
```

## 4. `orders.order_number` 생성 규칙

`order_number`가 `NOT NULL + UNIQUE`이므로 주문 생성 시 반드시 값을 넣어야 함.

**권장 생성 패턴:**
```
yyyyMMdd-HHmmss-{랜덤4자리}
예: 20260328-143052-A7K2
```

```java
public String generateOrderNumber() {
    String datePart = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMdd-HHmmss"));
    String randomPart = RandomStringUtils.randomAlphanumeric(4).toUpperCase();
    return datePart + "-" + randomPart;
}
```

**주의:** 동시 요청 시 중복 가능성이 있으므로 UNIQUE 제약 위반 시 재생성 로직 필요.

## 5. `orders.final_payment_id` 결제 흐름

순환 참조(orders ↔ payments) 구조이므로 **트랜잭션 내에서** 처리해야 함.

```java
@Transactional
public void processPayment(Long ordersId, PaymentRequest request) {
    // 1. payments INSERT (status = PENDING)
    Payment payment = paymentRepository.save(new Payment(ordersId, request));

    // 2. PG사 결제 요청
    PgResult result = pgClient.requestPayment(request);

    if (result.isSuccess()) {
        // 3. payments UPDATE (status = COMPLETE)
        payment.complete();

        // 4. orders.final_payment_id UPDATE
        Order order = orderRepository.findById(ordersId);
        order.setFinalPaymentId(payment.getPaymentId());
    } else {
        // 3. payments UPDATE (status = FAILED)
        payment.fail();
        // orders.final_payment_id는 NULL 유지
    }
}
```

**절대 금지:** `final_payment_id`를 payments INSERT와 동시에 세팅하지 말 것. 결제 성공이 확인된 후에만 UPDATE.

## 6. `sale_events.is_live` 제거됨 - VIEW 사용

`is_live` 컬럼이 제거되고 `live_sale_events` VIEW로 대체됨.

```java
// 라이브 세일 조회 시
@Query(value = "SELECT * FROM live_sale_events WHERE is_live = true", nativeQuery = true)
List<SaleEvent> findLiveSales();

// 또는 JPQL에서 직접 시간 비교
@Query("SELECT s FROM SaleEvent s WHERE CURRENT_TIMESTAMP BETWEEN s.saleStartAt AND s.saleEndAt")
List<SaleEvent> findLiveSales();
```

**주의:** JPA Entity는 `sale_events` 테이블에 매핑하되, `is_live` 필드는 Entity에 포함하지 않음. 필요하면 DTO에서 계산.

## 7. `deliveries` 생성 시 필수 데이터

`receiver_name`, `receiver_phone`, `receiver_address`, `delivery_status` 모두 NOT NULL.

```java
// delivery 생성 시 shipping_addresses에서 복사
public Delivery createFromShippingAddress(Order order, ShippingAddress addr) {
    return Delivery.builder()
        .ordersId(order.getOrdersId())
        .receiverName(addr.getReceiverName())       // 필수
        .receiverPhone(addr.getReceiverPhone())     // 필수
        .receiverAddress(addr.getAddress() + " " + addr.getAddressDetail())  // 필수
        .deliveryRequest(addr.getDeliveryRequest()) // 선택
        .deliveryStatus(DeliveryStatus.PREPARING)   // DEFAULT지만 명시 권장
        .build();
}
```

## 8. `cart_items` 중복 방지

`(user_id, product_id)` UNIQUE 제약이 있으므로 같은 상품을 카트에 두 번 넣으면 에러.

```java
// 카트 담기 전 존재 여부 확인
public CartItem addToCart(Long userId, Long productId) {
    Optional<CartItem> existing = cartItemRepository.findByUserIdAndProductId(userId, productId);
    if (existing.isPresent()) {
        throw new DuplicateCartItemException("이미 카트에 담긴 상품입니다.");
        // 또는 수량 개념이 없으므로 그냥 무시하고 기존 항목 반환
    }
    return cartItemRepository.save(new CartItem(userId, productId));
}
```

## 9. ENUM 값 매핑

DB ENUM과 Java Enum이 정확히 일치해야 함. 불일치 시 런타임 에러.

```java
// PostgreSQL ENUM과 1:1 매핑
public enum OrderStatus {
    PENDING, PAID, CANCELLED, REFUNDED
}

// JPA에서 문자열로 매핑
@Enumerated(EnumType.STRING)
@Column(name = "order_status")
private OrderStatus orderStatus;
```

**주의:** ENUM 값 추가/변경 시 DB ALTER와 Java Enum 수정을 반드시 동시에 배포할 것.

```sql
-- ENUM 값 추가 시 (PostgreSQL)
ALTER TYPE order_status_enum ADD VALUE 'SHIPPED';
-- 주의: ENUM 값 삭제는 PostgreSQL에서 직접 지원하지 않음
```

## 10. `payments` 금액 컬럼 일관성

`total_product_price`, `delivery_fee`, `discount_price`, `total_payment_price` 모두 `DEFAULT 0`.

**백엔드에서 반드시 검증:**
```java
// 결제 금액 계산 검증
public void validatePaymentAmount(Payment payment) {
    int expected = payment.getTotalProductPrice()
                 + payment.getDeliveryFee()
                 - payment.getDiscountPrice();

    if (expected != payment.getTotalPaymentPrice()) {
        throw new PaymentAmountMismatchException(
            "계산된 금액(" + expected + ")과 총 결제금액(" + payment.getTotalPaymentPrice() + ")이 불일치"
        );
    }

    if (payment.getTotalPaymentPrice() < 0) {
        throw new InvalidPaymentAmountException("결제 금액은 0 이상이어야 합니다.");
    }
}
```
