## 1. CHECK 제약조건 전무 (데이터 무결성)

DB 레벨에서 값의 유효 범위를 검증하는 CHECK 제약이 하나도 없음. 백엔드 버그나 직접 SQL 실행 시 비정상 데이터가 들어갈 수 있음.

| 테이블 | 컬럼 | 필요한 CHECK |
|--------|------|-------------|
| `products` | `price` | `price >= 0` |
| `payments` | `total_product_price` | `>= 0` |
| `payments` | `delivery_fee` | `>= 0` |
| `payments` | `discount_price` | `>= 0` |
| `payments` | `total_payment_price` | `>= 0` |
| `sale_events` | `sale_start_at`, `sale_end_at` | `sale_end_at > sale_start_at` |
| `coupons` | `expired_at` | `expired_at > created_at` |
| `product_images` | `sort_order` | `sort_order >= 0` |

```sql
-- 예시
ALTER TABLE "products" ADD CONSTRAINT "chk_products_price" CHECK ("price" >= 0);
ALTER TABLE "sale_events" ADD CONSTRAINT "chk_sale_events_dates" CHECK ("sale_end_at" > "sale_start_at");
ALTER TABLE "payments" ADD CONSTRAINT "chk_payments_amounts" CHECK (
    "total_product_price" >= 0 AND "delivery_fee" >= 0 AND
    "discount_price" >= 0 AND "total_payment_price" >= 0
);
```

## 2. `order_items.product_name` VARCHAR(50) vs `products.product_name` VARCHAR(100)

`order_items.product_name`은 주문 당시 스냅샷인데 길이가 50으로 원본(100)의 절반.
상품명이 50자를 초과하면 스냅샷 INSERT 시 에러 발생.

```sql
-- 현재
"product_name" VARCHAR(50) NULL   -- order_items
"product_name" VARCHAR(100) NOT NULL  -- products

-- 수정 필요
"product_name" VARCHAR(100) NULL  -- order_items도 100으로 맞춤
```

## 3. `final_payment_id` 인덱스 누락

`orders.final_payment_id`는 FK이지만 인덱스가 생성되지 않았음. 다른 FK 컬럼에는 모두 인덱스가 있는데 이것만 빠져있음.

```sql
CREATE INDEX "idx_orders_final_payment_id" ON "orders"("final_payment_id");
```

## 4. `shipping_addresses.is_default` 다중 기본 배송지 방지 불가

한 사용자가 여러 배송지에 `is_default = TRUE`를 설정할 수 있음. DB 레벨에서 "사용자당 기본 배송지 1개" 제약이 없음.

```sql
-- 부분 UNIQUE 인덱스로 해결
CREATE UNIQUE INDEX "uq_shipping_one_default_per_user"
ON "shipping_addresses"("user_id") WHERE ("is_default" = TRUE);
```

---

## 5. `product_images.is_main` 다중 대표 이미지 방지 불가

4번과 동일한 문제. 한 상품에 `is_main = TRUE`인 이미지가 여러 개 존재할 수 있음.

```sql
CREATE UNIQUE INDEX "uq_product_images_one_main"
ON "product_images"("product_id") WHERE ("is_main" = TRUE);
```

## 7. `sale_events`, `coupon_masters` 에 `updated_at` 없음

두 테이블 모두 수정 가능한 데이터(이벤트명 변경, 쿠폰 비활성화 등)인데 수정 시점을 추적할 수 없음.

```sql
-- sale_events에 추가
"updated_at" TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP

-- coupon_masters에 추가
"updated_at" TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
```

## 8. 중복 인덱스 우려 - `cart_items`

`cart_items`에는 다음이 존재:
- `uq_cart_user_product` UNIQUE (`user_id`, `product_id`) -- 복합 인덱스
- `idx_cart_items_user_id` (`user_id`) -- 단일 인덱스

UNIQUE 복합 인덱스의 선행 컬럼이 `user_id`이므로 `idx_cart_items_user_id`는 중복. 제거해도 무방.

```sql
-- 제거 가능
DROP INDEX "idx_cart_items_user_id";
-- uq_cart_user_product이 user_id 단독 조회도 커버함
```

> `idx_cart_items_product_id`는 product_id 단독 조회를 위해 필요하므로 유지.

## 9. `deliveries.orders_id` UNIQUE 제약 부재

한 주문에 배송 레코드가 여러 개 생성될 수 있음. 부분 배송을 지원한다면 의도된 것이지만, 그렇지 않다면 1:1 관계를 보장해야 함.

```sql
-- 1:1 관계가 맞다면
ALTER TABLE "deliveries" ADD CONSTRAINT "uq_deliveries_orders_id" UNIQUE ("orders_id");
```

## 10. `live_sale_events` VIEW 성능 우려

```sql
CREATE VIEW "live_sale_events" AS
SELECT *,
    (CURRENT_TIMESTAMP BETWEEN "sale_start_at" AND "sale_end_at") AS "is_live"
FROM "sale_events";
```

`SELECT *`를 사용하고 있어 모든 세일 이벤트를 스캔한 뒤 시간 비교를 수행. 데이터가 많아지면 느려짐.

```sql
-- 라이브 세일만 빠르게 조회하려면 인덱스 추가
CREATE INDEX "idx_sale_events_period" ON "sale_events"("sale_start_at", "sale_end_at");
```
