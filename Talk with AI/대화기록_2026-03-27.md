### 잘 된 부분
- Java 21 + Spring Boot 3.4.x 조합 — 최신이고 적절함

- JWT secret을 `${JWT_SECRET}` 환경변수로 관리 — 보안 측면 좋음
- Docker Compose (PostgreSQL + Redis + App) — 이미 구현됨
- `.env.example` 제공 — 협업 시 편리
- `GlobalExceptionHandler` + `CustomException` + `ErrorCode` Enum 구조 — 업계 표준
- `ApiResponse<T>` 공통 응답 구조 — 명확하고 일관성 있음
- Git Flow 브랜치 전략 (main → develop → feature/) — 적절

## Q4. 설계서 검토 보고서 피드백 및 반영 요청

**질문 요약**
> 검토 보고서 19개 항목에 대한 결정 사항 전달 및 추가 설명 요청

**주요 결정 사항**

| # | 항목 | 결정 |
|---|---|---|
| 1 | ddl-auto | DDL SQL 직접 작성으로 테이블 생성 |
| 2 | Flyway | 불필요 (Docker + 수동 DDL로 대체) |
| 3 | Spring Mail | 보류 (JavaMailSender vs Firebase 결정 후 추가) |
| 4 | 소셜 로그인 | 카카오 OAuth2 확정 |
| 6 | Payment | 별도 도메인 분리, Toss/카카오페이/네이버페이 연동 |
| 7,8 | review/wishlist | 불필요 (1회성 인플루언서 플리마켓 특성) |
| 10 | PageResponse | 불필요 (상품 10개 이하, 단일 페이지) |
| 12 | 파일 업로드 | AWS S3 확정 |
| 14 | influencer.user-id: 1 | 정상 설계 (인플루언서 1명) |
| 15 | hotfix 브랜치 | 불필요 |
| 19 | Docker | Dockerfile + docker-compose.yml 설계서에 추가 요청 |

**답변 요약**

아래 파일들을 모두 업데이트하여 피드백을 반영함:
- `MyFave_프로젝트_설계서.md` — 전면 재작성 (payment 도메인 추가, OAuth2/S3/결제 설정, Docker 섹션, API 버전 관리 등)
- `backend/build.gradle` — OAuth2, S3, WebFlux 의존성 추가
- `.env.example` — 카카오, S3, 결제사 환경변수 추가
- `backend/src/main/resources/application.yml` — context-path `/api/v1` 추가
- `backend/src/main/resources/application-local.yml` — 카카오 OAuth2, S3 설정 추가

**개념 설명 (초보자 대상)**

- **Flyway 불필요 이유**: Docker 볼륨으로 DB가 유지되고, DDL SQL로 직접 테이블을 관리하므로 마이그레이션 도구 없이도 가능
- **UserDetailsServiceImpl**: JWT 필터 ↔ UserRepository를 연결하는 다리. 없으면 Spring Security 인증이 동작하지 않음
- **BaseEntity**: 모든 엔티티의 `createdAt`, `updatedAt` 중복 제거. JPA Auditing으로 자동 저장
- **ApiResponse.error()**: GlobalExceptionHandler에서 일관된 에러 응답 형식 유지를 위해 필요
- **content ErrorCode**: `CustomException`을 던지려면 해당 도메인의 ErrorCode가 반드시 있어야 함
- **API 버전 관리**: `context-path: /api/v1` 설정 한 줄로 전체 API에 prefix 자동 적용
- **create-drop 주의**: 앱 재시작 시 데이터 전부 삭제됨. 개발 초기엔 편리하나 구조 안정 후 `create`로 변경 권장
- **staging 환경**: 소규모 팀은 local → prod 바로 배포도 일반적. 필요 시 나중에 추가

## Q5. DDL 검토 요청 (2026-03-28)

**질문**
> 14개 테이블 DDL 제공 후 프로젝트에 맞게 작성됐는지 검토 요청

**답변 요약**

| 구분 | 건수 |
|---|---|
| 🔴 치명적 오류 (실행 불가) | 8건 |
| 🟡 설계 오류 (기능 문제) | 6건 |
| 🟢 명명 일관성 문제 | 6건 |

**🔴 치명적 오류**
1. **MySQL 문법(백틱)** 사용 — 프로젝트는 PostgreSQL. 전체 재작성 필요
2. `TIMESTMAP` 오타 — deliveries(4곳), PAYMENT(2곳)
3. `category_code VARCHAR(20` — 닫는 `)` 없음 (문법 에러)
4. `ENUM` 값 미지정 (5곳) — PostgreSQL은 타입 선언 또는 CHECK 제약조건 필요
5. `TINYINT` — PostgreSQL에 없음. `BOOLEAN`으로 변경
6. `Key` 예약어 사용 (3곳) — delivery_id, short_form_id, style_feed_id로 변경
7. FK 제약조건 전부 누락
8. PK AUTO INCREMENT 없음 — `BIGSERIAL` 또는 `GENERATED ALWAYS AS IDENTITY` 필요

**🟡 설계 오류**
- `order_items`에 `quantity` 없음
- PAYMENT 테이블 PK/FK 명칭 혼동 + 외부 결제사 키 컬럼 없음
- `short_forms`에 `product_id` 중복
- `users`에 카카오 OAuth2 컬럼 없음 (`login_type`, `kakao_id`, password NULL 허용)
- `chat_messages` 테이블 없음
- `products`에 재고(`stock`) 컬럼 없음

**🟢 명명 일관성**
- SALE_ID, Coupon_master_id, user_id2, product_id2, PAYMENT → 모두 snake_case로 통일 필요
