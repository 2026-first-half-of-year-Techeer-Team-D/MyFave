## Q1. DDL 피드백 세부 질문

**질문**
> 1. ENUM 값 지정해야 하는 곳(5곳)이 어딘지 알려줘
> 2. FK 제약조건 전부 누락된 곳이 어딘지 알려줘
> 3. PK AUTO INCREMENT 없다는게 무슨말인지 알려줘
> 4. 상품당 수량은 전부 1개여서 order_items에 quantity는 없어도 돼
> 5. chat_messages 테이블이 왜 필요해?
> 6. 상품의 수량은 1개여서 재고컬럼은 필요 없어.

**답변 요약**

**ENUM 5곳:**
- `deliveries.delivery_status` → PREPARING, SHIPPED, DELIVERED, CANCELLED
- `orders.order_type` → 의도에 따라 결정
- `orders.order_status` → PENDING, PAID, CANCELLED, REFUNDED
- `PAYMENT.payment_method` → TOSS, KAKAO_PAY, NAVER_PAY
- `PAYMENT.payment_status` → PENDING, COMPLETED, FAILED, CANCELLED
- PostgreSQL에서는 `CREATE TYPE ... AS ENUM (...)` 선언 후 사용하거나 VARCHAR + CHECK 제약조건으로 대체

**FK 누락 16곳:**
deliveries→orders, orders→users, cart_items→users/products, shipping_addresses→users,
coupons→coupon_masters/users, order_items→products/orders, product_images→products,
PAYMENT→orders, chat_rooms→users, short_forms→products, style_feeds→products,
products→users/sale_events

**PK AUTO INCREMENT:**
- 현재 BIGINT로만 선언 → INSERT 시 ID를 직접 입력해야 함
- PostgreSQL에서는 `BIGSERIAL`로 선언하면 DB가 자동으로 1, 2, 3... 부여
- JPA에서 `@GeneratedValue(strategy = GenerationType.IDENTITY)` 사용 시 DDL도 BIGSERIAL이어야 함

**확정 사항:**
- `order_items.quantity` 불필요 — 상품당 1개 고정 ✅
- `products.stock` 불필요 — 동일 이유 ✅

**chat_messages 필요 이유:**
- WebSocket은 현재 연결된 사람에게만 전달. 나중에 입장한 사람은 이전 메시지 못 봄
- 이전 메시지 N개 표시, 채팅 로그 보존, 도배 방지(rate limit) 구현에 필요
- 최소 구조: message_id, chat_room_id(FK), user_id(FK), content, created_at
